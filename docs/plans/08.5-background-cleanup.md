# Plan 08.5 — Background Cleanup

Status: implemented.

## Goal

Remove fake or opaque backgrounds from generated PNGs without relying on external tools. This fills the gap between `prep alpha` (which only cleans existing transparency) and downstream commands like `slice` and `segment`, which work best once the subject is separated from the background.

The first version ships two techniques:

- `key`: remove an explicit keyed color such as magenta
- `edge`: infer the background from the image corners and remove only border-connected pixels that match it within a tolerance

The command is designed so additional techniques can be added later without changing the CLI shape.

## Scope

**In:**
- `internal/background` package with method dispatch and reusable color-matching helpers
- `sprite-gen prep background PATH`
- `--method auto|key|edge`
- `--color #RRGGBB` for explicit keying
- `--tolerance N` for both methods
- `--connectivity 4|8` for edge-connected flood fill
- Tests with synthetic PNG fixtures

**Out:**
- Semantic subject extraction / model-based cutout
- Despill, matte decontamination, and edge recoloring
- Interior hole filling beyond edge-connected removal
- Automatic checkerboard pattern detection
- Folding background removal directly into `segment subjects` in v1

## Why this slot in the sequencing

| # | Plan | Why `prep background` sits here |
|---|---|---|
| 06 | snap | Introduced alpha cleanup primitives, but only for images that already have meaningful transparency. |
| 07 | slice | Assumes the sheet background is already separated well enough to crop cells cleanly. |
| 08 | segment | Handles alpha-separated messy canvases, but opaque fake backgrounds still need a prior cleanup step. |
| **08.5** | **background cleanup (this plan)** | **Adds the missing one-image cleanup command for fake transparency and opaque generated backgrounds.** |
| 09 | align + diff | Still operates on frame sets only; background cleanup belongs upstream. |

This is intentionally a `prep` command, not part of `align`/`diff`: it transforms one source image into one cleaned image and composes naturally before either `slice` or `segment`.

## Package design

### `internal/background/background.go`

```go
package background

type Method string

const (
    MethodAuto Method = "auto"
    MethodKey  Method = "key"
    MethodEdge Method = "edge"
)

type Options struct {
    Method       Method
    KeyColor     color.NRGBA
    HasKeyColor  bool
    Tolerance    uint8
    Connectivity int // 4 or 8
}

type Result struct {
    Image         *image.NRGBA
    Method        Method
    RemovedPixels int
    ChangedPixels int
}

func Remove(img *image.NRGBA, opts Options) (Result, error)
```

`Remove` resolves `MethodAuto` to:

- `key` when `HasKeyColor == true`
- `edge` otherwise

The command surface stays stable even as new methods are added later.

### `internal/background/key.go`

```go
// remove all pixels whose RGB values are within tolerance of the keyed color
func removeKey(img *image.NRGBA, opts Options) (Result, error)
```

Rules:

- compare RGB only against `--color`
- set matching pixels to fully transparent
- preserve already-transparent pixels unchanged
- this is intentionally explicit and potentially destructive; callers opt in by choosing `--method key`

### `internal/background/edge.go`

```go
// sample corner colors and flood-fill inward from the image border,
// removing only connected pixels that stay within tolerance of one of
// the sampled background colors
func removeEdge(img *image.NRGBA, opts Options) (Result, error)
```

Algorithm:

1. Sample the four corner pixels as candidate background seeds.
2. Start a flood fill from every border pixel that matches one of those seeds.
3. Traverse using 4-connectivity by default (`--connectivity 8` is the escape hatch).
4. Remove only visited pixels; disconnected interior regions of the same color are preserved.

This makes `edge` a safer default than global keying for generated art where the subject may legitimately reuse the background color internally.

### `internal/background/color.go`

```go
func ParseHexColor(s string) (color.NRGBA, error)
func WithinTolerance(a, b color.NRGBA, tolerance uint8) bool
```

Use per-channel absolute difference for v1. It is easier to reason about and easier to explain in error messages than a more opaque distance metric.

## Command design

### `sprite-gen prep background PATH`

Reads one PNG and writes a cleaned PNG with the background removed.

Flags:

| Flag | Default | Notes |
|---|---|---|
| `--method auto\|key\|edge` | `auto` | `auto` chooses `key` when `--color` is supplied, otherwise `edge`. |
| `--color #RRGGBB` | unset | Required for `--method key`. Ignored by `edge`. |
| `--tolerance N` | `12` | Per-channel RGB(A) match threshold. |
| `--connectivity 4\|8` | `4` | Flood-fill neighborhood for `edge`. |
| `--out FILE` | `./out/<subject>/prep/background.png` | Output PNG path. |
| `--dry-run` | `false` | Report the output path without writing. |
| `--json` | global | `{ok, data, error}` envelope. |

Behavior:

1. Load the image with `pixel.LoadPNG`.
2. Parse and validate flags.
3. Dispatch to `background.Remove`.
4. Save the result.

Text output:

```
wrote: out/knight/prep/background.png
method: edge
removed_pixels: 73218
changed_pixels: 73218
```

JSON output:

```json
{
  "ok": true,
  "data": {
    "out": "out/knight/prep/background.png",
    "method": "edge",
    "removed_pixels": 73218,
    "changed_pixels": 73218,
    "tolerance": 12,
    "connectivity": 4,
    "dry_run": false
  }
}
```

For `--method key`, include the parsed key color in JSON so agents can verify the right input was applied.

Error surface:

- `--method key` without `--color` → `missing required --color for prep background --method key`
- invalid color string → actionable parse error naming the bad value
- invalid connectivity → `invalid --connectivity 5; want 4 or 8`

## Interaction with existing commands

- `prep alpha` remains the explicit cleanup step for real transparency with low-alpha haze.
- `prep background` removes fake or opaque backgrounds first.
- Many messy generated canvases will benefit from both:

```bash
sprite-gen prep background messy.png --method auto
sprite-gen prep alpha out/messy/prep/background.png --alpha-threshold 128
sprite-gen segment subjects out/messy/prep/clean.png --cell 32x32 --expected 4
```

The two prep commands are intentionally separate because they solve different problems and should stay individually composable.

## Testing

`internal/background/background_test.go`:
- `key` removes an exact flat magenta background and preserves the subject.
- `key` with `tolerance=12` removes near-magenta variations.
- `edge` removes a flat border-connected opaque background.
- `edge` preserves an enclosed interior region of the same color when it is not border-connected.
- `edge` with connectivity `8` can traverse diagonally where connectivity `4` cannot.
- `Remove(..., MethodAuto)` resolves to `key` when `HasKeyColor` is true and `edge` otherwise.

Command tests in `cmd/sprite-gen/cmd_prep_test.go`:
- `prep background img.png --method key --color '#FF00FF' --json` → envelope with `method: key` and `removed_pixels > 0`.
- `prep background img.png --method edge --out out.png` writes a PNG whose background pixels become transparent.
- `prep background ... --dry-run` writes nothing.
- `prep background --method key` without `--color` exits non-zero with an actionable message.
- `prep background --method edge --connectivity 8 --json` succeeds and reports the chosen connectivity.

## Acceptance criteria

1. `go test ./...` passes.
2. `sprite-gen prep background` removes a flat opaque keyed background without external tools.
3. `sprite-gen prep background --method edge` removes border-connected background pixels while preserving disconnected interior same-color regions.
4. `sprite-gen spec` shows `prep background`.
5. `README.md` explains when to use `prep background` vs `prep alpha`.
6. No new non-stdlib dependencies.
